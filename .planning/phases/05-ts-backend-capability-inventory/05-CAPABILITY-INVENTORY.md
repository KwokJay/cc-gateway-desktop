# Phase 05 Capability Inventory

This artifact is the canonical, source-cited inventory of the legacy TypeScript backend capabilities that matter to the standalone bootstrap CLI milestone. It exists so Phase 6 through Phase 9 can plan against the real checked-in TypeScript, script, config, and test surface without re-reading the full legacy implementation.

## Classification Rules

- `must-port`: The later standalone CLI phases need an equivalent behavior or contract to satisfy the local bootstrap, runtime-preparation, or `claude` launch requirements in Phase 6 through Phase 9.
- `reference-only`: The behavior is still planning-critical because later phases must preserve or understand it, but the milestone does not need to re-implement that capability in the new CLI path.
- `deferred`: The behavior exists in the legacy TS surface, but it belongs to remote-admin, multi-client, or broader deployment flows that the current local bootstrap milestone does not need to absorb in Phase 6 through Phase 9.
- Source precedence for this inventory is code, scripts, config, and tests over README narrative when they disagree.
- Source citations must stay at the behavior-summary level only. Do not copy raw tokens, credential payloads, or local secret values into this document.

## Known Drift

- OAuth refresh timing drift: `README.md` says the gateway auto-refreshes "5 minutes before expiry," but `src/oauth.ts` refreshes at actual expiry with `Math.max(msUntilExpiry, 1_000)`, and `tests/oauth.test.ts` explicitly rejects the old 5-minute-buffer behavior. Planning for Phase 6 through Phase 9 should treat the code and tests as the live contract.
- CCH and billing-header narrative drift: `README.md` says the billing header is stripped entirely and describes the CCH hash implementation as a fallback "not active by default," but `src/rewriter.ts` still computes CCH from the first rewritten user message and rewrites `cc_version=...` inside remaining system prompt text after stripping pure billing-header blocks; `docs/rust-ts-parity-checklist.md` also records TS parity work around active CCH handling.
- Config test coverage drift: `package.json` defines `npm test` as `tsx tests/rewriter.test.ts && tsx tests/oauth.test.ts`, so `tests/config.test.ts` is real validation coverage that sits outside the default test command. Phase 9 validation planning should keep that gap explicit instead of assuming `npm test` covers config semantics.
